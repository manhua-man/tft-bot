# Claude Code 大脑架构 — 完整调度与推理机制

> 基于 Claude-Code-rev 源码分析 (TypeScript, ~200K+ 行)

---

## 0. 一句话总结

```
Claude Code 是一个"QueryEngine + 51 工具 + 88 命令 + 5 层压缩 + 权限系统 + 协调者模式"的工程化系统。
它不只是一个 AI, 它是一个完整的应用框架。
```

## 3. 查询循环 — query() 函数的完整流程

```
┌─────────────────────────────────────────────────────────────────┐
│                    query() 函数流程 (~1700 行)                   │
│                                                                 │
│  用户输入                                                       │
│     │                                                           │
│     v                                                           │
│  ┌──────────────────────────────────────────┐                   │
│  │  Step 1: 构建 system prompt              │                   │
│  │                                           │                   │
│  │  fetchSystemPromptParts():                │                   │
│  │    getSystemPrompt()     ← 基础身份+规则 │                   │
│  │    getUserContext()       ← 用户上下文    │                   │
│  │    getSystemContext()     ← 系统上下文    │                   │
│  │    loadMemoryPrompt()    ← CLAUDE.md     │                   │
│  │    coordinator context   ← 协调者模式    │                   │
│  │    proactive suggestions ← 主动建议      │                   │
│  │                                           │                   │
│  │  组装:                                    │                   │
│  │    systemPrompt = defaultSystemPrompt     │                   │
│  │      + coordinatorUserContext             │                   │
│  │      + memoryPrompt                       │                   │
│  │      + appendSystemPrompt                 │                   │
│  └──────────────┬───────────────────────────┘                   │
│                 │                                                │
│                 v                                                │
│  ┌──────────────────────────────────────────┐                   │
│  │  Step 2: 准备消息                         │                   │
│  │                                           │                   │
│  │  2.1 获取附件消息:                        │                   │
│  │    getAttachmentMessages()                │                   │
│  │    - 记忆相关附件                         │                   │
│  │    - 文件附件                             │                   │
│  │                                           │                   │
│  │  2.2 预取:                                │                   │
│  │    startRelevantMemoryPrefetch()          │                   │

│  │    - 异步预取相关技能                     │                   │

│  │  2.3 去重:                                │                   │
│  │    filterDuplicateMemoryAttachments()     │                   │
│  │    - 移除重复的记忆附件                   │                   │
│  │                                           │                   │
│  │  2.4 标准化:                              │                   │
│  │    normalizeMessagesForAPI()              │                   │
│  │    - 格式化为 API 要求的格式              │                   │
│  │                                           │                   │
│  │  2.5 队列命令处理:                        │                   │
│  │    getCommandsByMaxPriority()             │                   │
│  │    - 处理排队的斜杠命令                   │                   │
│  │                                           │                   │
│  └──────────────┬───────────────────────────┘                   │
│                 │                                                │
│                 v                                                │
│  ┌──────────────────────────────────────────┐                   │
│  │  Step 3: 调用 Claude API                  │                   │
│  │                                           │                   │
│  │  query(system, messages, tools, model):    │                   │
│  │    - 携带完整 system prompt               │                   │
│  │    - 携带对话历史                         │                   │
│  │    - 携带 51+ 工具定义                    │                   │
│  │    - 流式接收响应                         │                   │
│  │    - thinkingConfig (思考模式)            │                   │
│  │    - maxTurns / maxBudgetUsd 限制         │                   │
│  │                                           │                   │
│  │  withRetry:                               │                   │
│  │    - 自动重试失败的 API 调用              │                   │
│  │    - FallbackTriggeredError → 降级模型    │                   │
│  │    - categorizeRetryableAPIError           │                   │
│  └──────────────┬───────────────────────────┘                   │
│                 │                                                │
│                 v                                                │
│  ┌──────────────────────────────────────────┐                   │
│  │  Step 4: 处理响应                         │                   │
│  │                                           │                   │
│  │  LLM 返回的内容块:                        │                   │
│  │                                           │                   │
│  │  ┌─ type: "text"                          │                   │
│  │  │  → 直接输出给用户                      │                   │
│  │  │                                        │                   │
│  │  ├─ type: "thinking"                      │                   │
│  │  │  → 显示思考过程                        │                   │
│  │  │                                        │                   │
│  │  ├─ type: "redacted_thinking"             │                   │
│  │  │  → 加密的思考 (安全原因)               │                   │
│  │  │                                        │                   │
│  │  ├─ type: "tool_use"                      │                   │
│  │  │  → 触发 runTools() 执行工具            │                   │
│  │  │  → 结果注入 messages                   │                   │
│  │  │  → 回到 Step 3 继续循环                │                   │
│  │  │                                        │                   │
│  │  └─ stop_reason: "end_turn"               │                   │
│  │     → 结束 turn                           │                   │
│  └──────────────┬───────────────────────────┘                   │
│                 │                                                │
│                 v                                                │
│  ┌──────────────────────────────────────────┐                   │
│  │  Step 5: 循环终止检查                     │                   │
│  │                                           │                   │
│  │  终止条件:                                │                   │
│  │  - end_turn (无更多工具调用)              │                   │
│  │  - 超过 maxTurns                          │                   │
│  │  - 超过 maxBudgetUsd                      │                   │
│  │  - 用户中断 (abort)                       │                   │
│  │  - 错误 (超过重试次数)                    │                   │
│  │  - prompt too long (触发压缩重试)         │                   │
│  │                                           │                   │
│  │  未终止 → 回到 Step 3 继续                │                   │
│  │  已终止 → 输出最终结果                    │                   │
│  └──────────────────────────────────────────┘                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 4. 工具编排 — runTools() 的并行/串行调度

```
┌─────────────────────────────────────────────────────────────────┐
│              toolOrchestration.ts — 工具并行调度                 │
│                                                                 │
│  这是 Claude Code 最精妙的设计。                                │
│                                                                 │
│  输入: LLM 返回的一组 ToolUseBlock                              │
│  输出: AsyncGenerator<MessageUpdate>                            │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  Step 1: partitionToolCalls() — 智能分区                │    │
│  │                                                         │    │
│  │  遍历每个 tool_use, 判断是否 isConcurrencySafe:         │    │
│  │                                                         │    │
│  │  ┌─────────────────────────────────────────────────┐    │    │
│  │  │  tool.isConcurrencySafe(input)                  │    │    │
│  │  │                                                 │    │    │
│  │  │  判断逻辑:                                      │    │    │
│  │  │  1. tool.inputSchema.safeParse(input)           │    │    │
│  │  │  2. tool.isConcurrencySafe(input)               │    │    │
│  │  │  3. 只读操作 → true (可并行)                    │    │    │
│  │  │  4. 写操作 → false (必须串行)                   │    │    │
│  │  └─────────────────────────────────────────────────┘    │    │
│  │                                                         │    │
│  │  分区结果:                                              │    │
│  │    batch 1: [FileRead, Grep]      → safe (并行)        │    │
│  │    batch 2: [FileEdit]            → NOT safe (串行)     │    │
│  │    batch 3: [FileRead]            → safe (并行)        │    │
│  │    batch 4: [Bash]                → NOT safe (串行)     │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  Step 2: 执行                                           │    │
│  │                                                         │    │
│  │  for each batch:                                        │    │
│  │                                                         │    │
│  │    if (isConcurrencySafe):                              │    │
│  │      → runToolsConcurrently()                           │    │
│  │      → 最多 MAX_TOOL_USE_CONCURRENCY (默认 10) 并行    │    │
│  │      → 等所有并行工具完成                               │    │
│  │      → 应用 context modifiers                           │    │
│  │                                                         │    │
│  │    else:                                                │    │
│  │      → runToolsSerially()                               │    │
│  │      → 一个接一个执行                                   │    │
│  │      → 每个完成后更新 context                           │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  Step 3: 上下文修改                                     │    │
│  │                                                         │    │
│  │  某些工具执行后会修改 ToolUseContext:                    │    │
│  │    - file_edit → 更新文件缓存                          │    │
│  │    - bash → 可能改变 cwd                               │    │
│  │    - 这些修改排队, batch 完成后统一应用                 │    │
│  │                                                         │    │
│  │  queuedContextModifiers: {                              │    │
│  │    [toolUseID]: ((ctx) => ctx)[]                        │    │
│  │  }                                                      │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  完整示例:                                                      │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  LLM 返回:                                              │    │
│  │    tool_use: FileRead("a.rs")                           │    │
│  │    tool_use: FileRead("b.rs")                           │    │
│  │    tool_use: FileEdit("a.rs", old, new)                 │    │
│  │    tool_use: Bash("cargo check")                        │    │
│  │                                                         │    │
│  │  分区:                                                  │    │
│  │    batch 1: [FileRead("a.rs"), FileRead("b.rs")] → 并行│    │
│  │    batch 2: [FileEdit("a.rs")] → 串行                  │    │
│  │    batch 3: [Bash("cargo check")] → 串行               │    │
│  │                                                         │    │
│  │  执行:                                                  │    │
│  │    并行读取 a.rs 和 b.rs                                │    │
│  │    → 完成后 → 串行编辑 a.rs                            │    │
│  │    → 完成后 → 串行执行 cargo check                     │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```
