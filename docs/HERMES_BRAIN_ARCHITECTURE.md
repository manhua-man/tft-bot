# Hermes Agent 大脑架构 — 完整调度与推理机制

---

## 0. 一句话总结

```
Hermes 不是"一个聪明的 AI"。
它是一个"5 层注入 + 16 工具 + 83 技能 + 护栏 + 压缩 + 记忆"的系统。
AI 只是这个系统里的推理引擎。
```

## 2. "先扫描技能" — 强制知识注入

```

因为 system prompt 里有硬性指令。

┌─────────────────────────────────────────────────────────────────┐
│                    技能扫描流程                                  │
│                                                                 │
│  用户输入: "帮我写 Rust 代码，用 ort crate 加载 ONNX 模型"     │
│                                                                 │
│  Step 1: 扫描 available_skills 列表                             │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  rust-python-rl-bot: RL 游戏机器人                       │    │
│  │  rust-python-rl-pipeline: RL 训练管线                    │    │
│  │  rust-workspace-migration: Rust 工作空间迁移             │    │
│  │  github-pr-workflow: PR 全生命周期                       │    │
│  │  ...                                                      │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  Step 2: 匹配 → 加载 rust-python-rl-bot                         │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  skill_view("rust-python-rl-bot")                        │    │
│  │                                                          │    │
│  │  加载内容:                                                │    │
│  │  ┌──────────────────────────────────────────────────┐    │    │
│  │  │  ## Pitfalls                                      │    │    │
│  │  │  - ort 2.0 的 Tensor::from_array 不接受          │    │    │
│  │  │    (shape, slice), 要用 (shape, Box<[f32]>)      │    │    │
│  │  │  - Session::run 需要 &mut self, 不是 &self       │    │    │
│  │  │  - Windows 上 include_str! 路径用 ../../../       │    │    │
│  │  │    不是 ../../ (从 src/ 到 workspace root)        │    │    │
│  │  │  - ort::inputs! 宏用位置参数, 不是命名参数       │    │    │
│  │  │  - outputs[0].try_extract_tensor 返回 (Shape,     │    │    │
│  │  │    &[f32]), 不是直接返回 &[f32]                   │    │    │
│  │  └──────────────────────────────────────────────────┘    │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  Step 3: 推理时参考这些知识                                     │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  我写代码时:                                              │    │
│  │    Tensor::from_array(([1, 34], obs_vec.into_boxed_slice())) │
│  │    而不是:                                                │    │
│  │    Tensor::from_array(([1, 34], obs_vec.as_slice()))     │    │
│  │                                                          │    │
│  │    session.run(ort::inputs![input])                      │    │
│  │    而不是:                                                │    │
│  │    session.run(ort::inputs!["observation" => input])     │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  效果: 第一次就写对, 不需要编译→报错→修复的循环                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 3. Tool Call → Observe → Decide 循环 — 推理的"心跳"

```
每个 turn 的执行模式。这不是线性的, 是循环的。

┌─────────────────────────────────────────────────────────────────┐
│                    推理循环 (一个 turn)                          │
│                                                                 │
│  用户输入                                                       │
│     │                                                           │
│     v                                                           │
│  ┌──────────────────────────────────────────┐                   │
│  │  思考 (推理)                              │                   │
│  │                                           │                   │
│  │  我分析:                                  │                   │
│  │  - 用户想要什么                           │                   │
│  │  - 当前状态是什么                         │                   │
│  │  - 需要哪些步骤                           │                   │
│  │  - 哪些步骤可以并行                       │                   │
│  └──────────────┬───────────────────────────┘                   │
│                 │                                                │
│                 v                                                │
│  ┌──────────────────────────────────────────┐                   │
│  │  决策: 选哪种执行模式                     │                   │
│  │                                           │                   │
│  │  ┌─────────────────────────────────────┐  │                   │
│  │  │  模式 A: 单工具调用                  │  │                   │
│  │  │  适用: 一个明确操作                  │  │                   │
│  │  │  例子: read_file("Cargo.toml")      │  │                   │
│  │  └─────────────────────────────────────┘  │                   │
│  │                                           │                   │
│  │  ┌─────────────────────────────────────┐  │                   │
│  │  │  模式 B: 多工具并行                  │  │                   │
│  │  │  适用: 多个无依赖操作                │  │                   │
│  │  │  例子:                               │  │                   │
│  │  │    search_files("*.rs") +            │  │                   │
│  │  │    read_file("Cargo.toml") +         │  │                   │
│  │  │    terminal("git status")            │  │                   │
│  │  │  → 同一 function block 并行发出      │  │                   │
│  │  └─────────────────────────────────────┘  │                   │
│  │                                           │                   │
│  │  ┌─────────────────────────────────────┐  │                   │
│  │  │  模式 C: 代码块                      │  │                   │
│  │  │  适用: 复杂逻辑、循环、条件判断      │  │                   │
│  │  │  例子:                               │  │                   │
│  │  │    execute_code {                    │  │                   │
│  │  │      for file in files:              │  │                   │
│  │  │        result = read_file(file)      │  │                   │
│  │  │        if "error" in result:         │  │                   │
│  │  │          ...                         │  │                   │
│  │  │    }                                 │  │                   │
│  │  │  → 最多 50 次工具调用, 5 分钟超时    │  │                   │
│  │  └─────────────────────────────────────┘  │                   │
│  │                                           │                   │
│  │  ┌─────────────────────────────────────┐  │                   │
│  │  │  模式 D: 子代理                      │  │                   │
│  │  │  适用: 复杂独立子任务                │  │                   │
│  │  │  例子:                               │  │                   │
│  │  │    delegate_task({                   │  │                   │
│  │  │      goal: "创建 tft-executor crate", │  │                   │
│  │  │      context: "...",                 │  │                   │
│  │  │      toolsets: ["terminal", "file"]  │  │                   │
│  │  │    })                                │  │                   │
│  │  │  → 子代理独立执行, 返回 summary      │  │                   │
│  │  └─────────────────────────────────────┘  │                   │
│  └──────────────────────────────────────────┘                   │
│                 │                                                │
│                 v                                                │
│  ┌──────────────────────────────────────────┐                   │
│  │  执行工具                                 │                   │
│  │                                           │                   │
│  │  工具返回结果:                            │                   │
│  │  - stdout / stderr                        │                   │
│  │  - exit_code                              │                   │
│  │  - 文件内容                               │                   │
│  │  - 错误信息                               │                   │
│  └──────────────┬───────────────────────────┘                   │
│                 │                                                │
│                 v                                                │
│  ┌──────────────────────────────────────────┐                   │
│  │  观察结果 + 重新评估                      │                   │
│  │                                           │                   │
│  │  我看:                                    │                   │
│  │  - 命令成功了吗? (exit_code == 0?)       │                   │
│  │  - 输出符合预期吗?                        │                   │
│  │  - 有新的信息需要处理吗?                  │                   │
│  │  - 下一步该做什么?                        │                   │
│  │  - 任务完成了吗?                          │                   │
│  └──────────────┬───────────────────────────┘                   │
│                 │                                                │
│                 ├──── 未完成 ───→ 回到"思考"继续                │
│                 │                                                │
│                 └──── 完成 ────→ 输出最终回复                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

关键: 每次工具返回后, 我重新评估状态再决定下一步。
不是"写好计划然后盲执行"。
是"做一步, 看一步, 想一步, 再做下一步"。
```

## 4. 并行判断逻辑 — 我怎么知道哪些能同时做

```
┌─────────────────────────────────────────────────────────────────┐
│                    并行 vs 串行决策                              │
│                                                                 │

│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │

│  │                                                         │    │
│  │  例子 2:                                                │    │
│  │    terminal("cargo check -p tft-domain")  ──┐           │    │
│  │    terminal("cargo check -p tft-sim")     ──┼── 并行    │    │
│  │    terminal("cargo check -p tft-env")     ──┘           │    │
│  │    (三个 crate 编译互不影响)                            │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  有依赖 → 串行                                          │    │
│  │                                                         │    │
│  │  例子 1:                                                │    │
│  │    read_file("lib.rs")  →  看到内容  →  patch(...)      │    │
│  │    (patch 依赖 read_file 的结果)                        │    │
│  │                                                         │    │
│  │  例子 2:                                                │    │
│  │    cargo check  →  看到错误  →  修复代码  →  cargo test │    │
│  │    (每一步依赖前一步的结果)                             │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  混合: 部分并行 + 部分串行                              │    │
│  │                                                         │    │
│  │  例子:                                                  │    │
│  │    阶段 1 (并行):                                       │    │
│  │      read_file("lib.rs") + read_file("Cargo.toml")     │    │
│  │    阶段 2 (串行, 依赖阶段 1):                          │    │
│  │      patch("lib.rs", old, new)                         │    │
│  │    阶段 3 (并行, 依赖阶段 2):                          │    │
│  │      cargo check + cargo test                          │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  这不是我显式写的 if/else。                                     │
│  是我在思考时自然地把无依赖的放在一起,                          │
│  有依赖的等前面的完成后再决定。                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

│                                                                 │

│     │                                                           │
│     v                                                           │
│  ┌──────────────────────────────────────────┐                   │
│  │  失败 1: 分析错误信息                     │                   │
│  │                                           │                   │
│  │  我看:                                    │                   │
│  │  - exit_code 是什么?                      │                   │

│  │  - 是参数错误? 环境错误? 权限错误?        │                   │
│  │                                           │                   │
│  │  行动:                                    │                   │
│  │  - 修正参数, 重试                         │                   │
│  │  - 或: 换一种方法                         │                   │
│  └──────────────┬───────────────────────────┘                   │

│  │  - 用不同的工具                           │                   │
│  │  - 用不同的命令                           │                   │

│                 │                                                │
│                 v                                                │
│  ┌──────────────────────────────────────────┐                   │

## 8. Skill 的陷阱注入 — 过去踩过的坑不会再踩

```
┌─────────────────────────────────────────────────────────────────┐
│                    Skill 陷阱注入机制                            │
│                                                                 │
│  Skill 不只是"怎么做", 还有"会踩什么坑"。                      │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  rust-python-rl-bot skill 示例                          │    │
│  │                                                         │    │
│  │  ## Pitfalls                                            │    │
│  │                                                         │    │
│  │  1. ort crate API 差异                                  │    │
│  │     错误: Tensor::from_array(([1,34], slice))          │    │
│  │     正确: Tensor::from_array(([1,34], box_slice))      │    │
│  │     原因: ort 2.0 要求 Box<[f32]>, 不接受 &[f32]       │    │
│  │                                                         │    │
│  │  2. Session::run 可变性                                 │    │
│  │     错误: fn predict(&self, ...)                        │    │
│  │     正确: fn predict(&mut self, ...)                    │    │
│  │     原因: Session::run 需要 &mut self                   │    │
│  │                                                         │    │
│  │  3. include_str! 路径                                   │    │
│  │     错误: include_str!("../../configs/x.json")          │    │
│  │     正确: include_str!("../../../configs/x.json")       │    │
│  │     原因: 从 crates/xxx/src/ 到 workspace root 需要    │    │
│  │           三级 ../../../                                │    │
│  │                                                         │    │
│  │  4. ort::inputs! 宏                                     │    │
│  │     错误: ort::inputs!["name" => tensor]               │    │
│  │     正确: ort::inputs![tensor]                          │    │
│  │     原因: 旧 API 用位置参数, 不是命名参数              │    │
│  │                                                         │    │
│  │  5. try_extract_tensor 返回值                           │    │
│  │     错误: let data = output.try_extract_tensor::<f32>() │    │
│  │     正确: let (shape, data) =                           │    │
│  │           output.try_extract_tensor::<f32>()            │    │
│  │     原因: 返回元组 (Shape, &[f32])                      │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  效果:                                                          │
│  - 第一次写就写对                                              │
│  - 不需要 编译→报错→查文档→修复 的循环                         │
│  - 这些坑是之前实际踩过的, 写进 skill 后不再重复              │
│                                                                 │
│  对比没有 skill 的情况:                                        │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  没有 skill:                                            │    │
│  │    写代码 → 编译失败 → 查错误 → 修复 → 编译 → 失败     │    │

│  │    (3-5 轮循环)                                         │    │
│  │                                                         │    │
│  │  有 skill:                                              │    │
│  │    参考 pitfalls → 写代码 → 编译成功                    │    │
│  │    (1 轮)                                               │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 11. 本质: 不是 AI 聪明, 是系统设计好

```
┌─────────────────────────────────────────────────────────────────┐
│                    设计哲学                                      │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  1. 注入而非硬编码                                      │    │
│  │     System prompt 是动态拼装的, 不是静态字符串。         │    │
│  │     记忆、技能、项目上下文都按需注入。                   │    │
│  │     我不需要"猜", 因为信息已经在我面前。                │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  2. 工具而非命令                                        │    │
│  │     不靠拼接 shell 字符串, 而是用专用工具。              │    │
│  │     patch 代替 sed (9 种模糊匹配)                       │    │
│  │     read_file 代替 cat (行号、分页、语法检查)           │    │
│  │     write_file 代替 echo (自动建目录、覆盖保护)         │    │
│  │     这些工具封装了容易出错的边界情况。                   │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  3. 护栏而非信任                                        │    │
│  │     不假设 AI 不会犯错, 而是在架构层防止。               │    │
│  │     tool_loop_guardrails 强制停止死循环。                │    │
│  │     超时控制防止无限等待。                               │    │
│  │     输出截断防止内存溢出。                               │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  4. 隔离而非共享                                        │    │
│  │     子代理有独立上下文, 不污染父代理。                   │    │
│  │     后台进程有独立终端, 不阻塞主对话。                   │    │
│  │     定时任务有独立会话, 不影响当前会话。                 │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  5. 经验而非记忆                                        │    │
│  │     Skill 记录"踩过的坑", 不靠 AI 记忆。                │    │
│  │     每次加载最新版本, 自动修复过时内容。                 │    │
│  │     新发现的坑立即 patch 到 skill 里。                   │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  6. 重新评估而非盲目执行                                │    │
│  │     每次工具返回后, 我重新评估状态。                     │    │
│  │     不是"写好计划然后盲执行"。                           │    │
│  │     是"做一步, 看一步, 想一步, 再做下一步"。            │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  ═══════════════════════════════════════════════════════════    │
│                                                                 │
│  这 6 条加在一起, 效果就是"丝滑"。                             │
│                                                                 │
│  不是因为 AI 聪明。                                             │
│  是因为系统把容易出错的环节都封装了。                           │
│  AI 只需要决定"做什么", 工具层保证"怎么做"不翻车。             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```
